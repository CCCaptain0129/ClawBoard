# 任务看板使用说明与模板

> 规范化任务创建流程，提高团队协作效率

---

## 📋 任务模板

### 基础模板

```markdown
-  **TASK-{序号}** `{优先级}` `{标签1}` `{标签2}`
  - 状态: {状态}
  - 描述: {任务描述}
  - 负责人: {负责人}
```

### 完整示例

```markdown
-  **TASK-100** `P1` `frontend` `bugfix`
  - 状态: 待处理
  - 描述: 修复用户登录后无法显示头像的问题
  - 负责人: @username
```

---

## 🏷️ 字段说明

### 优先级 (Priority)

| 优先级 | 含义 | 使用场景 |
|--------|------|----------|
| `P1` | 高优先级 | 阻塞性问题、核心功能、紧急 bug |
| `P2` | 中优先级 | 常规功能、优化任务、一般 bug |
| `P3` | 低优先级 | 文档、测试、非紧急任务 |

**示例：**
- `P1` - 生产环境崩溃、关键功能失效
- `P2` - 新功能开发、性能优化
- `P3` - 文档更新、代码整理

### 状态 (Status)

| 状态 | 含义 | 自动切换方式 |
|------|------|--------------|
| `待处理` | 任务已创建，未开始 | 默认状态 |
| `进行中` | 正在开发 | 点击"开始 →"按钮 |
| `已完成` | 任务完成 | 点击"完成 ✓"按钮 |

### 标签 (Labels)

标签用于分类和筛选，常用标签：

**按领域分类：**
- `frontend` - 前端任务
- `backend` - 后端任务
- `devops` - 运维任务
- `docs` - 文档任务
- `test` - 测试任务
- `design` - 设计任务

**按类型分类：**
- `feature` - 新功能
- `bugfix` - Bug 修复
- `refactor` - 重构
- `optimization` - 优化
- `hotfix` - 紧急修复

**自定义标签：**
可以添加任意自定义标签，如：
- `auth` - 认证相关
- `ui` - UI 界面
- `api` - API 接口
- `database` - 数据库

**示例：**
- `P1` `frontend` `bugfix` - 高优先级前端 bug 修复
- `P2` `backend` `feature` `auth` - 中优先级后端认证功能
- `P3` `docs` - 低优先级文档任务

### 负责人 (Assignee)

格式：`@username` 或直接写用户名

**示例：**
- `@john`
- `@alice`
- `@team-frontend`

## 📝 创建任务最佳实践

### 1. 任务标题清晰明确

❌ 不好的例子：
- 修复问题
- 做个功能
- 改一下

✅ 好的例子：
- 修复用户登录后无法显示头像的问题
- 实现任务拖拽功能
- 优化首页加载速度（目标：< 1s）

### 2. 描述要具体

**包含信息：**
- 具体要做什么
- 预期结果是什么
- 依赖关系（如果有）

**示例：**
```markdown
-  **TASK-100** `P1` `frontend` `feature`
  - 状态: 待处理
  - 描述: 实现任务卡片拖拽功能，支持在不同状态列之间拖拽任务，拖拽完成后自动更新任务状态并同步到后端
  - 负责人: @john
```

### 3. 合理设置优先级

**P1（高优先级）场景：**
- 生产环境问题
- 阻塞其他任务的问题
- 用户反馈的核心问题

**P2（中优先级）场景：**
- 计划内的功能开发
- 性能优化
- 一般 bug 修复

**P3（低优先级）场景：**
- 文档更新
- 代码优化
- 非紧急改进

### 4. 使用标签分类

**组合标签：**
- `P1` `frontend` `bugfix` - 前端紧急 bug
- `P2` `backend` `feature` `auth` - 后端认证功能
- `P3` `docs` `api` - API 文档更新

## 🎯 不同类型任务模板

### Bug 修复

```markdown
-  **TASK-{序号}** `{优先级}` `{领域}` `bugfix`
  - 状态: 待处理
  - 描述: [问题简述] - [复现步骤] - [预期结果]
  - 负责人: @username
```

**示例：**
```markdown
-  **TASK-200** `P1` `frontend` `bugfix`
  - 状态: 待处理
  - 描述: 登录后头像无法显示 - 复现步骤：1. 登录系统 2. 查看右上角头像 - 预期结果：显示用户头像
  - 负责人: @john
```

```markdown
-  **TASK-{序号}** `{优先级}` `{领域}` `feature`
  - 状态: 待处理
  - 描述: [功能描述] - [核心需求] - [验收标准]
  - 负责人: @username
```

**示例：**
```markdown
-  **TASK-201** `P2` `frontend` `feature`
```

```markdown
-  **TASK-{序号}** `{优先级}` `{领域}` `optimization`

  - 描述: [优化目标] - [当前性能] - [目标性能]
```

## 🔄 任务工作流

### 标准流程

```
待处理 → 进行中 → 已完成
   ↓         ↓
  暂停      挂起
```

### 状态转换规则

| 当前状态 | 可转换状态 | 触发方式 |
|----------|------------|----------|
| 待处理 | 进行中 | 点击"开始 →" |
| 进行中 | 已完成 | 点击"完成 ✓" |
| 进行中 | 待处理 | 手动编辑（未实现） |
| 已完成 | 待处理 | 手动编辑（未实现） |

## 📊 任务编号规则

### 编号格式

```
TASK-{3位数字}
```

### 分段编号

按阶段或模块分配编号段：

```
TASK-001 ~ TASK-099  : 阶段 1
TASK-100 ~ TASK-199  : 阶段 2
TASK-200 ~ TASK-299  : 阶段 3
...
```

### 按类型编号

```
TASK-1XX  : Bug 修复
TASK-2XX  : 功能开发
TASK-3XX  : 优化任务
TASK-4XX  : 文档任务
TASK-5XX  : 测试任务
```

## 💡 高级技巧

### 1. 任务依赖

在描述中标注依赖关系：

```markdown
-  **TASK-100** `P2` `frontend` `feature`
  - 状态: 待处理
  - 描述: 实现任务拖拽功能
  - 依赖: TASK-099（拖拽库集成）
  - 负责人: @john
```

### 2. 子任务

使用编号后缀表示子任务：

```markdown
-  **TASK-100** `P2` `frontend` `feature`
  - 状态: 待处理
  - 描述: 实现任务拖拽功能
  - 负责人: @john

-  **TASK-100-1** `P2` `frontend`
  - 状态: 待处理
  - 描述: 集成拖拽库

-  **TASK-100-2** `P2` `frontend`
  - 状态: 待处理
  - 描述: 实现拖拽逻辑

-  **TASK-100-3** `P2` `frontend`
  - 状态: 待处理
  - 描述: 测试拖拽功能
```

### 3. 关联 Issue

在描述中添加 GitHub Issue 链接：

```markdown
-  **TASK-200** `P1` `frontend` `bugfix`
  - 状态: 待处理
  - 描述: 修复头像显示问题 - Issue: #123
  - 负责人: @john
```

## ⚠️ 常见错误

### 错误 1：缺少优先级

❌ 错误：
```markdown
-  **TASK-100** `frontend` `bugfix`
```

✅ 正确：
```markdown
-  **TASK-100** `P1` `frontend` `bugfix`
```

### 错误 2：描述过于简单

❌ 错误：
```markdown
-  **TASK-100** `P1` `frontend` `bugfix`
  - 状态: 待处理
  - 描述: 修复问题
```

✅ 正确：
```markdown
-  **TASK-100** `P1` `frontend` `bugfix`
  - 状态: 待处理
  - 描述: 修复用户登录后头像无法显示的问题
```

### 错误 3：标签过多

❌ 错误：
```markdown
-  **TASK-100** `P1` `frontend` `bugfix` `ui` `avatar` `urgent` `today`
```

✅ 正确：
```markdown
-  **TASK-100** `P1` `frontend` `bugfix`
```

## 🔧 技术细节

### Markdown 格式要求

1. **任务标题**：使用 `**TASK-{序号}**` 格式
2. **元数据**：同一行，用反引号包裹
3. **状态**：使用 `- 状态: ` 格式
4. **描述**：使用 `- 描述: ` 格式
5. **负责人**：使用 `- 负责人: ` 格式

### 解析规则

- 优先级、标签通过反引号 `` ` `` 识别
- 状态通过"状态: "识别
- 描述通过"描述: "识别
- 负责人通过"负责人: "识别
